Archiving-specific Changes

The Archiving Tag Set, archivearticle3.dtd, was brought up to Version 3.0, with the DTD version attribute (@dtd-version) set to “3.0 ” and the version and date changed in every module to “v3.0 20080202 ”.

Archiving Changes in Classes

The following changes were made to the base classes defined in archivecustom-classes3.ent:

  • Alternatives Clashes — There is sometimes a need to allow alternatives, but since DTDs cannot handle interleaving, there is also a grammar requirement to name only one <alternatives> element even when groups of display objects are combined. So, in default-classes3.ent, created the following element classes for use in combinations; they hold the contents of the like-named parameter entity, minus only the <alternatives> element:
    • %block-display-noalt.class; (used in archivecustom-models3.ent)
    • %simple-display-noalt.class; (used in archivecustom-models3.ent)
  • Citations — Created a new type of citation <element-citation> and added it to %citation.class;.
  • References Class — The Archiving Tag Set no longer overrides %references.class;, which is used in the models of <mixed-citation>, <element-citation>, <product>, <related-article>, and <related-object>.
  • In archivecustom-classes3.ent, removed %references.class; as duplicative of the Suite default.
    In default-classes3.ent, in %references.class;, added the elements:
    • <chapter-title>,
    • <part-title>,
    • <date-in-citation>,
    • <size>,
    • <string-name>, and
    • <volume-series>.

    Also deleted the following elements from %references.class;:
    • <access-date> (replaced by <date-in-citation>),
    • <page-count> (replaced by <size>), and
    • <time-stamp> (replaced by <date-in-citation>).

    The current default references class (used by all Tag Sets) is now:

  • Rest of Paragraph Class — Removed <x> from this class as <x>has now been added to both %all-phrase; and %para-level;.

Major Model Change — The element <x> was added to the following elements (using %x.class;) , resulting in major model changes:

  • Contributor — In archivecustom-models3.ent, changed <contrib> (using %contrib-model;) from a semi-sequential model that required the name elements to be first: 
    ( (%name.class;)*, 
(%degree.class; | %contrib-info.class; | 
%x.class;)* )

    to a large OR group which includes <x> and all the contributor information class elements as well as all of the name class elements in any order as required, which should give archives maximum freedom: 
    ( (%name.class; | %degree.class; | 
%contrib-info.class; | %x.class;)* )
  • Publication Date — In archivecustom-models3.ent, changed <pub-date> from a sequential model: 
    (season?, (day?, month?)?, year?, string-date?)

    to a large OR group which includes <x> as well as all the potential dates: 
    (day | month | year | season | string-date | x)*
  • Reference List — In archivecustom-models3.ent, changed a <ref-list> sequence of paragraph-level objects followed by all the references <ref>: 
    (title?, (%para-level;)*, ref*, (%ref-list.class;)* )

    to a large OR group, with <x> coming in through para-level and all paragraph objects being allowed to be interspersed with references (<ref>): 
    (label*, title?, (%para-level; | ref)*, 
(%ref-list.class;)* )

    Best practice would not tag interspersed paragraphs in a reference list, since these almost always indicate the presence of subsidiary (untitled) reference lists. In an authoring or publishing tag set, such lists would be identified explicitly. But archiving tag sets do not always have the luxury of reading and understanding the content being converted, and all that can be determined without human intelligence is that paragraphs are interspersed with references <ref> inside the reference list.

Where <x> is Not — By design, <x> was therefore not added to elements that were viewed as critical #PCDATA-only metadata such as identifiers, page numbers, and date components. The <x> was also not added to the few restricted, single-function elements in the Tag Sets (such as <glyph-data>). Therefore, the element <x> was explicitly not added to the following #PCDATA-only elements:

(contrib-group | aff | %x.class;)*, 
author-notes?, pub-date*,
volume?, volume-id*, volume-series?, 
issue?, issue-id*, issue-title*,
issue-sponsor*, issue-part?, isbn*, 
supplement?, 
( ( (fpage, lpage?)?, page-range?) | 
elocation-id )?, 
(%address-link.class; | product | 
supplementary-material)*, 
history?, permissions?, self-uri*, 
related-article*, abstract*, 
trans-abstract*, kwd-group*, funding-group*, 
conference*, counts?, custom-meta-group?)

The changes to make this new model (using %article-meta-model;) are:

  • Funding Information — Replaced the following element grouping with “funding-group*”: 
    contract-num*, contract-sponsor*, grant-num*, grant-sponsor*,
  • Loose Copyright — Inside %article-meta-model;, moved <copyright-statement>, <copyright-year>, and <license> to inside <permissions> instead of being both inside and outside it. The modeling: 
    history?, copyright-statement?, 
copyright-year?, license?, permissions?, self-uri*,

    was changed to: 
    history?, permissions?, self-uri*,
  • Issue Part — Added <issue-part> and <issue-sponsor> to <article-meta> following the other issue information: 
    issue?, issue-id*, issue-title*,
issue-sponsor*, issue-part?, isbn*,
  • ISBN — Added an optional, repeatable <isbn> as part of the potential issue information inside <article-meta>.
  • String Conference — Added new element <string-conf>, which uses (%string-conf-elements;) for use inside <conference> using %conference.class;. This element provides a way to record textual statements of all the conference information in one place. For best practice (when possible), this single statement should be in addition to, not as a replacement for, the regular conference metadata.
  • Volume Series — Added <volume-series> to <article-meta> to hold the series number, in those odd cases where, for example, a Journal has started reissuing with number one and has added a series number to the volume numbers: 
     volume?, volume-id*, volume-series?,

Archiving Other Article Metadata

Some elements within the article metadata also changed models:

  • Article Title — In archivecustom-models3.ent, added an override for the new parameter entity %article-title-elements; to be used inside <article-title> instead of %title-elements;, because the content of all structural titles was expanded, but the content for <article-title> should not do so. The <article-title> element now includes: 
    %all-phrase; | %break.class;
  • Author Notes — In archivecustom-models3.ent, expanded the model for <author-notes>, by adding the following elements to the OR group inside <author-notes> (using %author-notes-model;):
    • <p> (using %just-para.class;) (which can contain both text and, for example, <named-content>)
    • <x> (using %x.class;)
  • Collab — In archivecustom-models3.ent, enhanced the model of <collab> (using %collab-elements;) to allow additional elements inside a <collab>, to make it more like a contributor. The original model for <collab> included the following elements: 
    %all-phrase; | %break.class; | %x.class;
    • Added the elements of the %contrib-info.class; to this OR mix. That adds the following elements to <collab>: 
      address | aff | author-comment | bio | 
email | etal | ext-link | on-behalf-of |
role | uri | xref
    • Also added <contrib-group> to <collab>
    • Also added %address.class;, which added the elements: 
      addr-line | country | fax | institution | phone

    (Implementor’s Note: When %contrib-info.class; was added to <collab>, that caused some duplication errors. Changed the content of %collab-elements; from including %all-phrase; to including most of the pieces of %all-phrase;. Added %contrib-info.class; to the mix, which necessitated removing %address-link.class;, %simple-link.class;, and %fn-link.class; to get rid of duplications.)
  • Contributor Model — In archivecustom-models3.ent, modified the model of <contrib> ( using %contrib-model;) from an ordered model where the name elements must precede all other elements, to a optional repeatable OR group that includes <x>. 
    (%name.class; | %degree.class; | %contrib-info.class; | %x.class;)*
  • Contrib Group — The new %contrib-group.class; adds <contrib-group> to <collab> (using the override %collab-elements;).
  • Keyword Group — In archivecustom-models3.ent, modified the model of <kwd-group> ( using %kwd-group-model;):
    • Added an optional <label> before the <title>,
    • Allowed <x> as an alternative to <kwd>, and
    • Added <unstructured-kwd-group> as an alternative to a bunch of keywords.
    The resulting model is as follows: 
    (label*, title?,
( (%kwd.class; | %x.class;)+ | unstructured-kwd-group ) )
  • Publication Date — In archivecustom-models3.ent, modified the model of <pub-date> ( using %pub-date-model;) from an ordered model where season must precede day and month and both must precede year, to a optional repeatable OR group that includes <x> along with all the date elements, because complete freedom in dates is more in the spirit of Archiving: 
    (day | month | year | season | string-date | x)*
  • Related Object — In archivecustom-models3.ent, modified <related-object> by allowing the following elements to be mixed with text (overriding %related-object-elements;) . (Note: This includes the content named by %all-phrase;, which was not used explicitly because of duplication clashes with %references.class;):

Archiving Funding Information

  • Funding Statement —<funding-statement>. This is a mixed-content model patterned after the <copyright-statement>. In the Suite default, which all Tag Sets except Archiving use, the content includes the text, potentially mixed with emphasis elements, subscript and superscript, the phrase-level elements such as <named-content>, and a few links. In Archiving, the content includes the parameter entity %all-phrase;, which allows the new elements <award-id>, <funding-source>, and <open-access> to be mixed into the text, since they may not be identified separately.
  • Inside Article Metadata — In the module archivecustom-models3.ent, replaced the following element grouping by “funding-group*”: 
    contract-num*, contract-sponsor*, grant-num*, grant-sponsor*,
  • Inside a Paragraph — In archivecustom-models3.ent, modified Paragraph (<p> using %p-elements;) by replacing contract-grant.class (which named the old grant and contract elements) with %funding.class; (which names the new funding and award elements).
  • Award-ID Attributes — In the module archivecustom-models3.ent), the attributes for <award-id> are expanded to be the same optional attributes available for <award-group>, since <award-id>s are more likely to be loose in text in Archiving than in the other Tag Sets.
    • @id (an ID-type attribute, so the award can be referenced)
    • @rid (an IDREFS-type attribute)
    • @award-type (a textual description or categorization of the award)
    • and the usual linking attributes for optional linking

Archiving Front Stub Metadata

In archivearticle3.dtd, enhanced the <front-stub> model to include all elements in <article-meta>. Thus, in Archiving, the potential metadata elements are same for both <front-stub> and <article-meta>, to accommodate whatever subset a publisher might think appropriate for a <front-stub>, while retaining the convenience of not being required to repeat the infrastructure of a full <front> for a <sub-article> .

The Archiving Tag Set no longer has an override for the %references.class;. This Tag Set uses the default references elements, adding special Archive-only elements such as <x> to the individual models of the citation elements (<element-citation> and <mixed-citation>) using the parameter entity %citation-elements; .

Archiving Chemical Structures Changes

Although it was not the intent when the Tag Sets were written, most Tag Set users have been treating <chem-struct> as an inline and <chem-struct-wrap> as a block structure. This greatly simplifies all processing with the two and seems quite flexible, so steps were taken to regularize this usage:

  • As an inline object, <chem-struct> should be able to be used anywhere that <inline-formula> is used, therefore <chem-struct> has been added to %inline-math.class; (in default-classes3.ent.
  • As an inline object, <chem-struct> should not be used as a block-level object, so it was removed from %block-display.class; (in default-classes3.ent).
  • As an inline object, <chem-struct> must be inside <chem-struct-wrap> (using %chem-struct-wrap-model;) (in display3.ent).
  • As a block object, <chem-struct-wrap> needs to be everywhere block display objects are used, so it needs to be part of the %block-display.class; and %floats-display.class;, which it already was in the default (default-classes3.ent).
  • As a block object, <chem-struct-wrap> should be inside figures (using %fig-model;), which it already was in the default (display3.ent).
  • Changed the OR group allowed inside <table-wrap> (using %inside-table-wrap;) to allow <chem-struct-wrap> instead of <chem-struct>.

The element <chem-struct> has been removed from within the following elements, where it has been allowed in previous versions, allowing only the block <chem-struct-wrap> to be used: [Backward incompatible]

  • <table-wrap> (using %inside-table-wrap;)

Following these changes, <chem-struct> is now allowed within the following elements:

  • <article-title>
